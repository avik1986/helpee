2. System Architecture

Helpee is a modular monolith (NestJS) serving five client applications through a versioned REST API with WebSocket support for real-time features. PostgreSQL with PostGIS handles all persistent data including geospatial queries. Redis provides caching and pub/sub for live location streaming. BullMQ handles async jobs. AWS provides the cloud infrastructure.

Data Flow Diagrams

Care Request to Shift Activation Flow

Family creates request→System assesses→Matching Engine runs→Top 3 shared→Family selects→Agreement signed→Roster created→Shift scheduled→Caregiver notified

Shift Execution Flow

Shift scheduled → Caregiver notified → Navigates → Check-in (geofence) → Shift active → GPS 60s logging → Tasks + Vitals logged → Check-out (geofence) → Shift completed → Earnings calc → Summary to family

Payment Flow

Shifts in month → Invoice generated → Family notified → Razorpay payment → Webhook received → Invoice paid → Earnings updated → Payroll batch → Admin approval → Payouts (T+7)

Abnormal Vitals Alert Flow

Caregiver logs vital → Threshold check → Abnormal detected → Push to family (≤60s) + Push to admin → Alert logged → Admin acknowledges → Escalation if needed

7. Security Architecture

Layers

  • TLS 1.3 everywhere (client ↔ ALB ↔ services)
  • JWT auth flow: Login → Access Token (15min) + Refresh Token (7d) → Refresh rotation
  • RBAC: 5 roles with permission matrix below
  • Data encryption: AES-256 at rest (RDS, S3), TLS in transit
  • Health data segregation: vitals_log in separate schema/access controls
  • API rate limiting per role
  • Audit logging on all admin actions

8. Real-Time Architecture

Location Streaming

Caregiver app sends GPS every 60s → API → Redis pub/sub → Admin/Family clients poll or use WebSocket

Shift Status Updates

Status change → Redis publish → WebSocket broadcast to subscribed clients

Abnormal Vitals Alerts

Vitals logged → Threshold check → If abnormal → BullMQ job → Push (FCM) + In-app + WebSocket event

No-Show Detection

Cron every 5 min → Checks shifts where scheduled_start + 15min < now AND status = 'scheduled' → Marks no_show → Triggers alert

GPS 60s → API → Redis Pub/Sub → WebSocket / Poll
Vitals → Threshold → BullMQ → Push + WebSocket
Cron 5min → No-show check → Mark no_show → Alert

9. Scalability Strategy

  • Current: Modular monolith (single NestJS app) — fast development, simple deployment
  • Phase 1 extraction candidates: Location Tracking Service (high write volume), Notification Service (async, independent)
  • Horizontal scaling: ECS auto-scaling based on CPU/memory (target 70%)
  • Database: Read replicas for dashboard queries, connection pooling (PgBouncer)
  • Caching: Redis for hot data (active shifts, caregiver locations, dashboard metrics)
  • CDN: Static assets via CloudFront, API response caching for catalog endpoints
  • Target: 10K concurrent caregivers, 50K patients, ~500 RPS at peak

12. Sprint-by-Sprint Breakdown

Sprint 1 (Weeks 1–2): Foundation Setup

Phase 0 — Foundation

Goals: Monorepo scaffold, DB schema, all 5 app skeletons, CI/CD

  • Backend: Turborepo setup (apps/api, apps/company-admin, apps/franchise-admin, apps/caregiver, apps/customer, apps/elderly). PostgreSQL + PostGIS + Prisma schema (35+ tables). Base NestJS structure with modules, guards, interceptors, pagination.
  • Mobile: React Native Expo project scaffold for 3 mobile apps (caregiver, customer, elderly). Navigation structure, design system tokens, shared component library.
  • Admin: Next.js 14 project scaffold for both Company Admin and Franchise Admin portals. Shared layout system with sidebar, routing structure.
  • DevOps: Docker configs, GitHub Actions pipeline (lint, test, build), AWS staging environment.

Deliverables: Working monorepo, all 5 apps running locally, CI green, DB schema deployed to staging.

Sprint 2 (Weeks 3–4): Auth + RBAC + Infrastructure

Phase 0 — Foundation

Goals: Auth flow complete for all 5 roles, infra services operational

  • Backend: Auth module (OTP send/verify via MSG91, admin email+password login, JWT access+refresh, RBAC middleware for 5 roles: company_admin, franchise_admin, caregiver, family_user, elderly). Base middleware, rate limiting, error handling.
  • Mobile: OTP login screens for all 3 mobile apps. Design system implementation.
  • Admin: Login page for both admin portals with role-based redirect.
  • DevOps: S3 bucket creation, Redis setup (OTP store, cache), BullMQ queue infrastructure.

Deliverables: Full auth flow working for all 5 roles, infra services operational.

Sprint 3 (Weeks 5–6): Caregiver Onboarding

Phase 1 — P0 MVP

Goals: Caregiver registration, document upload, admin verification

  • Backend: Caregiver CRUD APIs, document upload to S3, verification workflow endpoints, contract type (direct/franchise) handling, status management (lead → eligible → bg_check → assessed).
  • Mobile (Caregiver): Profile setup wizard (name, phone, address, experience, skills), document upload screen (Aadhaar, PAN, certificates, photo), verification status tracking.
  • Company Admin: Caregiver list page with filters, caregiver detail view, document verification actions, direct CT onboarding flow.

Deliverables: Caregiver can register + upload docs, admin can verify and onboard.

Sprint 4 (Weeks 7–8): Care Request + Matching Engine

Phase 1 — P0 MVP

Goals: Family booking flow, algorithmic matching

  • Backend: Care request CRUD, matching engine (PostGIS proximity + skill_level + availability, weighted scoring: ratings, cancellation rate, acceptance speed, SOP adherence, complaint history), match result APIs (top 3), caregiver selection endpoint.
  • Mobile (Customer): Registration + elderly patient profile creation, care request form (service type, shift type, schedule, preferences), matched profiles screen (top 3 cards with photo, rating, experience).
  • Company Admin: Care request list with status filters, matching status view, manual override capabilities.

Deliverables: Full booking flow from request to top 3 profile matching.

Sprint 5 (Weeks 9–10): Selection + Agreement + Rostering

Phase 1 — P0 MVP

Goals: Caregiver selection, agreement signing, roster creation

  • Backend: Caregiver selection + lock endpoint, service agreement generation (PDF via S3), e-sign with OTP, roster CRUD with conflict detection, shift creation from roster, geofence configuration per patient.
  • Mobile (Customer): Caregiver selection flow, shortlist/compare, agreement view + e-sign, registration fee payment (Razorpay).
  • Mobile (Caregiver): Job offer notification, offer detail screen, accept/reject with reason.
  • Company Admin: Calendar view (week/month), roster creation form, drag-drop assignment.

Deliverables: Selection → agreement → roster flow working E2E.

Sprint 6 (Weeks 11–12): Shift Lifecycle + Geo-Attendance

Phase 1 — P0 MVP

Goals: Shift status transitions, geofence check-in/out, GPS tracking

  • Backend: Shift status transitions (scheduled → en_route → arrived → active → completed), geofence check-in (Haversine validation), GPS batch logging endpoint (60s intervals), no-show detection cron (every 5 min), deviation alerts.
  • Mobile (Caregiver): Home screen with upcoming shifts, navigation deeplink (Google Maps), check-in button with location permission, shift timeline view.
  • Mobile (Customer): Live shift tracking with status timeline, caregiver ETA, real-time map.
  • Company Admin: Attendance monitoring table, no-show alerts feed.

Deliverables: Full geo-attendance flow, live shift tracking working.

Sprint 7 (Weeks 13–14): Task Logging + Notifications + SOS

Phase 1 — P0 MVP | MILESTONE: P0 MVP BETA

Goals: Task checklist, daily summaries, push notifications, SOS

  • Backend: Task log CRUD, daily summary auto-generation, FCM push notification integration (shift updates, no-show, payment), SOS endpoint (triggers call + push to admin + family).
  • Mobile (Caregiver): Active shift screen with task checklist, mark tasks complete with timestamp + notes, SOS button (always visible).
  • Mobile (Customer): Daily summary view (tasks, notes), push notification handling.
  • Company Admin: Governance actions (blacklist, suspend), notification management.
  • QA: E2E test suite for core P0 flows (onboarding → booking → shift → summary).

Deliverables: P0 MVP Beta — core Caregiver + Customer + Company Admin flows working.

Sprint 8 (Weeks 15–16): Ledger + Payments + Invoicing

Phase 1 — P0 MVP

Goals: Earnings, invoicing, Razorpay integration

  • Backend: Earning calculation per completed shift, invoice generation (monthly), Razorpay integration (payment order creation, webhook handler), company ledger aggregation, approval workflow engine (multi-level chains).
  • Mobile (Customer): Invoice list, payment flow (Razorpay SDK), payment confirmation, receipt download.
  • Mobile (Caregiver): Earnings screen (shift-wise, weekly, monthly), payout status, earnings history.
  • Company Admin: Financial ledger (revenue by city, margin analysis), payout exports (CSV/Excel), approval queue, invoice generation status.

Deliverables: Payments working E2E, ledger + approvals live.

Sprint 9 (Weeks 17–18): Dispute Management + Company Admin Dashboard

Phase 1 — P0 MVP

Goals: Dispute system, national dashboard, admin polish

  • Backend: Dispute CRUD with resolution workflow, categorization (late arrival, quality, behavior, billing), evidence attachment to S3.
  • Mobile (Customer): Dispute creation form with category selection, evidence upload, resolution tracking.
  • Company Admin: National dashboard (active shifts with city breakdown, no-show alerts, supply gap indicators, quick actions, abnormal vitals feed), dispute queue with resolution workflow.
  • QA: Payment flow testing, dispute workflow testing, dashboard data validation.

Deliverables: Dispute system working, national dashboard live with real-time data.

Sprint 10 (Weeks 19–20): Franchise Admin — Core Setup

Phase 1 — P0 MVP

Goals: Franchise admin RBAC, city dashboard, caregiver pool

  • Backend: Franchise admin scoping (all queries filtered by franchise_id), franchise caregiver pool endpoints, franchise-specific attendance endpoints.
  • Franchise Admin: City dashboard (active shifts, roster overview, supply gap, pending actions, key metrics), caregiver pool management (view CTs with status/availability/certification/ratings, filter, initiate onboarding, availability calendar).
  • QA: RBAC testing — verify franchise admins can only see their own city data, data isolation checks.

Deliverables: Franchise admin portal operational with dashboard + CT pool management.

Sprint 11 (Weeks 21–22): Franchise Admin — Rostering + Attendance

Phase 1 — P0 MVP

Goals: Local rostering, attendance monitoring for franchise

  • Backend: Franchise-specific roster endpoints, local conflict detection, franchise attendance log endpoints.
  • Franchise Admin: Local rostering (assign CTs to care requests, drag-drop calendar, bulk schedule, quick-swap replacement, conflict detection, auto-suggest best-fit CT), attendance monitoring (real-time check-in/out log with geofence validation, no-show alerts, geo-violation alerts, late arrival tracking).
  • Company Admin: Rostering oversight enhancements (global view across franchises), emergency reassignment.

Deliverables: Franchise can independently manage local rostering and monitor attendance.
